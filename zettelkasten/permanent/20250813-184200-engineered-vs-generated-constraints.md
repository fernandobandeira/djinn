# Engineered vs. Model-Generated Constraints: Value Framework

## Core Insight
There are two fundamental approaches to probability shaping in LLM interactions: externally engineered constraints (molecular context, examples, structure) and internally model-generated constraints ("thinking" models). Each approach provides different value propositions: engineered constraints offer precision and control, while generated constraints offer adaptability and context-specific optimization.

## Evidence & Examples
### Concrete Example
**Engineered**: Providing specific code examples to constrain programming solutions
**Generated**: Letting a thinking model work through the problem internally to generate its own constraint patterns
**Hybrid**: Providing seed examples and letting the model expand with internal constraints

### Abstract Pattern
The choice between external precision vs. internal adaptability represents a classic engineering trade-off: control vs. flexibility, predictability vs. optimization.

### Edge Cases
Some contexts may require engineered constraints (high precision, known patterns), while others benefit from generated constraints (novel problems, context-specific optimization).

## Value Analysis Framework
### When Engineered Constraints Excel
- High precision requirements
- Known optimal patterns
- Predictable contexts
- Resource constraints (token limits)
- Reusable scenarios

### When Generated Constraints Excel
- Novel or complex problems
- Context-specific optimization needed
- Unpredictable problem variations
- When model has superior domain knowledge
- Adaptive optimization scenarios
- **When constraint requirements are unclear upfront**
- Exploratory problem-solving phases
- When you need the model to discover optimal constraint patterns

### When Hybrid Approaches Excel
- Complex multi-faceted problems
- When both precision and adaptability are needed
- Resource-rich environments
- Learning/improvement scenarios
